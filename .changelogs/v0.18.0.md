# Release v0.18.0 - Accumulate Mode Calculation Fixes

Released: 2026-01-13

## Overview

This release fixes critical calculation bugs in accumulate mode for funds with partial sells. The invested amount and expected target were being calculated incorrectly, leading to misleading gain percentages and recommendations.

## 🐛 Bug Fixes

### Liquidation Detection
- **Fixed liquidation detection to use multiple methods**: Now uses OR logic combining share-based, value-based, and dollar-based detection
- Previously, only share-based detection (`cumShares < 0.0001`) was used, which failed to detect liquidations when small dust amounts remained
- Value-based detection: `remainingValue <= sellAmount + 0.01`
- Dollar-based detection: `totalSells >= totalBuys`
- Applied consistently across engine, FundCharts, and backtest simulator

### Invested Amount Calculation (Accumulate Mode)
- **Fixed invested amount resetting on full liquidation**: The 2023-11-09 liquidation for robinhood-btc was not being detected, causing invested to show ~$969K instead of the correct ~$768K
- Invested now correctly represents sum of buys since the last full liquidation

### Expected Target Calculation (Accumulate Mode)
- **Fixed expected gain not being reduced by sells**: In accumulate mode, expected gain is based purely on invested principal compounding at target APY
- Partial sells do NOT reduce expected gain (they are profit extraction, not position reduction)
- Only full liquidations reset the expected gain
- Fixed both engine (`computeExpectedTarget`) and client-side chart calculation (`FundCharts.tsx`)

## 🔧 Improvements

### Consistency Across Codebase
- Unified liquidation detection logic across:
  - `packages/engine/src/expected-equity.ts` (computeStartInput, computeExpectedTarget)
  - `packages/web/src/components/FundCharts.tsx` (chart time series)
  - `pages/src/engine/backtest.ts` (backtest simulator)
- All three locations now use identical multi-method liquidation detection

## 📊 Impact Example (robinhood-btc)
